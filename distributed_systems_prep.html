CAP theorem & consistency models

The CAP theorem states a distributed system can guarantee at most two of three properties simultaneously. PACELC extends this with latency tradeoffs for the normal (non-partition) case.

CAP theorem
Consistency, availability, partition tolerance
Core

C — Consistency: every read returns the most recent write, or you get an error. Think of it as: the system looks like a single node — all replicas agree. Important nuance: in CAP, 'consistent' specifically means linearizability, not the everyday English word.

A — Availability: every request gets a non-error response, though that response might be stale.

P — Partition tolerance: the system keeps running even when nodes can't communicate with each other.

The key insight: network partitions are inevitable in real distributed systems, so P is non-negotiable — you always have it. The real choice is what you do during a partition: stay consistent and reject some requests (CP), or stay available and risk returning stale data (AP). CP examples: HBase, ZooKeeper, Spanner. AP examples: Cassandra, DynamoDB, CouchDB.

Interview framing: CAP is a binary, partition-only model — don't say "pick two" glibly. The honest answer is "P is non-negotiable, so I'm choosing how the system behaves during a partition." Then pivot to PACELC for the normal-case latency tradeoff.

PACELC
Extends CAP with latency tradeoffs
Advanced

CAP only tells you what happens during a partition. But most of the time your system isn't partitioned — so what then? PACELC answers this: even in the normal case, you're still trading off latency against consistency. Being consistent requires coordination across nodes, and coordination takes time.

During Partition: choose A or C. Else (normal operation): choose L (low latency) or C (consistency).

Linearizability
Strongest consistency — the C in CAP
Strict

The mental model: the system behaves as if there's exactly one copy of the data, and every operation takes effect at a single instant in real time. The moment a write completes, every read anywhere in the system must see it.

The real-time constraint is what makes it so strong. If my operation finishes before yours starts, you must observe my effect. That's what separates it from sequential consistency — which only requires some consistent ordering, but doesn't have to match actual wall-clock time.

To achieve this, all reads have to go through the leader — or you use lease-based reads to prevent a stale leader from serving old data. You need linearizability for distributed locks and leader election: anywhere correctness truly depends on seeing the latest value. Used by etcd, ZooKeeper, Spanner.

Don't confuse the two "atomicity"s: the "A" in ACID (a transaction is all-or-nothing) is unrelated to atomic/linearizable consistency (single-copy, real-time order). Interviewers love poking at this overload.

Eventual consistency
AP systems converge over time
Common

If no new writes occur, all replicas will converge to the same value — but reads may return stale data in the meantime. There's no guarantee on when convergence happens.

This is what you get when you sacrifice the C in CAP — it's not on the same scale as linearizability, it's the alternative guarantee offered in its place.

Variants: Read-your-writes — you always see your own writes. Monotonic reads — reads never go backward in time. Causal consistency — causally related ops are seen in order.

Causal consistency
Between eventual and linearizable
Nuanced

The idea: if event A causally influenced event B, every node must see A before B. But concurrent events — things that happened with no causal relationship — can be seen in any order. It sits between eventual consistency and linearizability in strength.

Classic example: you post a comment after seeing a photo. Anyone who sees your comment must also see the photo, because your comment was causally caused by it. Causality is tracked with vector clocks or version vectors.

This is often the right trade-off for collaborative and social applications. And it turns out to be the strongest consistency model you can achieve while staying fully available during a partition — linearizability isn't achievable under those conditions.

Why it matters: causal consistency is the strongest model achievable while staying fully available during partitions (the "Consistency, Availability, Convergence" / COPS result). Linearizability is not — that's the practical ceiling for an AP system.

CRDTs
Conflict-free replicated data types
Convergence

The problem CRDTs solve: when two replicas independently accept concurrent writes, how do you merge them without conflicts and without any coordination? CRDTs are data structures specifically designed so that any two replicas that've seen the same set of updates always converge to the same state — regardless of what order they saw them in, with no coordination needed.

Two flavours: State-based — replicas exchange their full state and merge it; the merge must be commutative, associative, and idempotent (a semilattice). Operation-based — replicas broadcast the operations themselves, which must be commutative; requires reliable causal delivery.

Common types to know: G-Counter / PN-Counter for distributed counters; G-Set / OR-Set for sets where add wins over remove; LWW-Register for last-write-wins; sequence CRDTs like RGA for collaborative text editing.

G-Counter — 3 nodes A, B, C, each tracks its own increments. Two replicas diverged during a partition:

Replication

Keeping copies of data on multiple nodes for fault tolerance and read scalability. The core tradeoff: synchronous replication = durable but slow; asynchronous = fast but risks data loss.

Single-leader replication
One writer, many readers
Most common

One leader takes all writes and propagates them to followers via a replication log. Reads can go to followers — potentially stale — or to the leader for a consistent read.

Sync vs async: Synchronous means the leader waits for at least one follower to acknowledge before confirming the write — more durable, but write latency roughly doubles. Asynchronous means the leader replies immediately and replicates in the background — fast, but if the leader crashes before a follower catches up, those writes are lost.

The sneaky lag issues: read-after-write inconsistency (you write to the leader but immediately read from a lagging follower, so you don't see your own write); monotonic read violations (two reads hit different followers at different lag points — time seems to go backwards); causality violations across clients. These are the bugs that are hard to reproduce in testing but very real in production.

Multi-leader replication
Multiple writers, conflict resolution needed
Complex

Each datacenter has its own leader. Writes go to the local leader, then replicate asynchronously to remote leaders. Better write performance for multi-region deployments.

Conflict resolution: Last Write Wins (LWW) — fast but silently drops data when clocks disagree. Merge (CRDTs) — mathematically safe, limited to certain data types. On-write — application logic rejects the conflicting write. On-read — return all versions to the application.

LWW is dangerous: concurrent writes from different regions with clock skew can silently discard the higher-value write.

Chain replication
Ordered chain: strong consistency + throughput
Internals

The idea: replicas form an ordered chain, and writes flow one direction through it. All writes go to the head, propagate node by node, and only get acknowledged once the tail confirms. All reads go to the tail. Because the tail only sees fully-propagated writes, reads are linearizable.

Plain chain: ack only after the tail has the write; linearizable reads go to the tail. CRAQ: clean nodes answer reads directly; dirty nodes ask the tail for the committed value.

This naturally splits load: the head handles write ingestion, the tail handles reads — better throughput than routing everything through one leader. Failure recovery is clean too — head, tail, and middle nodes each have a well-defined recovery rule. CRAQ (Chain Replication with Apportioned Queries) extends this so any node can serve reads. A clean node answers directly; a dirty node asks the tail for the committed version. You get read scaling without sacrificing linearizability.

Anti-entropy
Background reconciliation via Merkle trees
Convergence

Anti-entropy is a background process that continuously compares replicas and syncs any differences. It ensures eventual convergence even for keys that nobody is actively reading — as opposed to read repair, which only fixes things when a read happens to trigger a comparison.

Merkle trees make this efficient. Hash each key+value at the leaves, then hash upward so each parent hashes its children. To compare two replicas, just compare root hashes. If they match, done — the replicas are identical. If not, descend the tree to pinpoint exactly which key range diverged and sync only that. That's O(log N) to find the divergence instead of O(N) comparing everything.

Consensus

Getting multiple nodes to agree on a value even when some fail. Fundamental to leader election, distributed locks, and state machine replication.

What consensus is used for
One primitive — "agree on a value" — wearing many hats
Mental model

Consensus is a single primitive: get a quorum of nodes to agree on one value despite failures. A surprising number of distributed-systems problems are just that primitive in disguise — each is "agree on X":

  • Strong-consistency replication (SMR) → agree on the order of operations (the replicated log) ⇒ linearizability.
  • Leader election → agree on which node leads this term — e.g. electing a shard's single leader.
  • Cluster membership → agree on who is in the group (Raft joint consensus for adding/removing nodes).
  • Distributed locks / leases → agree on who holds the lock (Chubby, ZooKeeper).
  • Config / metadata store → agree on a key's value — where the partition/slot map lives (etcd and ZooKeeper are consensus systems); the very map request routing reads.
  • Fault-tolerant atomic commit → make the 2PC coordinator a consensus group so it can't block (Spanner runs 2PC across Paxos groups for cross-shard transactions).
The punchline: leader election is not a separate problem from consistency — it's the same problem. "Agree on who the leader is" and "agree on the next log entry" both reduce to "get a quorum to agree on one value." That's why Raft bundles leader election and log replication into one protocol — election comes essentially for free.
It's usually leader-based (Raft / Multi-Paxos / ZAB), and it's exactly what leaderless-quorum systems (Dynamo, Cassandra) deliberately skip to buy availability — which is why they're eventually consistent.

Terms: monotonically increasing numbers, one per election cycle. Every message carries a term. If a node sees a higher term than its own, it immediately updates and steps down if it was a leader. Messages with a lower term are rejected.

Election: followers wait for heartbeats. If none arrives within a randomised timeout (150–300ms), a follower increments its term, declares itself a candidate, votes for itself, and sends RequestVote to everyone. It wins if it gets a strict majority. The crucial constraint: a node won't vote for a candidate whose log is less up-to-date than its own. This guarantees the elected leader always has all committed entries.

Log up-to-date check (lexicographic): 1. Higher lastLogTerm wins 2. If tied, higher lastLogIndex wins → guarantees the winner holds every *committed* entry (it may still be missing a prior leader's *uncommitted* ones)

Log replication: the leader appends the entry locally, sends AppendEntries to all followers in parallel, and commits once a majority acknowledges. Each AppendEntries includes prevLogIndex + prevLogTerm so the follower can verify log consistency — if it doesn't match, it rejects, and the leader walks back until it finds the common point.

Commit rule: a leader can only commit entries from its current term by majority. Previous-term entries are committed implicitly when a current-term entry commits and drags them along. This avoids a subtle bug where a replicated-but-uncommitted entry from a previous term gets overwritten by a new leader.

Paxos
Original consensus algorithm
Classic

Phase 1 — Prepare: the proposer picks a ballot number n and sends Prepare(n) to a majority. Each acceptor promises not to accept any lower ballot, and reports back the highest-ballot value it's already accepted — if any.

Phase 2 — Accept: the proposer picks a value v — either its own proposal, or the highest-ballot accepted value it heard back in Phase 1 (it must use that if one exists). It sends Accept(n, v). Acceptors accept if they haven't seen a higher ballot since they promised.

That's single-decree Paxos — one agreed value. Multi-Paxos extends it for log replication: elect a stable leader and skip Phase 1 for subsequent entries, amortising the election cost across many writes. The reason most new systems use Raft instead: Paxos has many tricky edge cases around leader crashes between phases, and the original paper famously leaves out the implementation details.

ZAB & ZooKeeper
Atomic broadcast for coordination
Coordination

ZAB (ZooKeeper Atomic Broadcast) is the consensus protocol underneath ZooKeeper. The idea is similar to Raft — leader-based, total-order broadcast — but it's specifically designed around recovering a consistent log prefix after a leader change.

zxid: every transaction gets a 64-bit ID structured as epoch:counter. The epoch increments each time a new leader is elected; the counter increments per write. This gives a global total order and makes it unambiguous which leader issued which write — and it naturally doubles as a fencing token for distributed locks.

What you actually build with ZooKeeper: it exposes a hierarchical key-value store (znodes) with watches (notify me when this path changes), ephemeral nodes (auto-deleted when the session that created them dies), and sequential nodes. From just these primitives you can implement leader election, distributed locks, config management, and service discovery.

Reads are not linearizable by default: ZooKeeper serves reads from any follower for throughput, so they can be stale. Call sync() before a read, or use the leader, when you need a fresh read. Writes always go through the leader and are linearizable.

Reads in consensus systems
The stale leader problem
Critical

Here's a subtle point that trips people up: even routing reads to the leader isn't automatically linearizable. A partitioned old leader might not know a new one was elected, and keeps serving stale reads. That's a linearizability violation.

Leader leases: the leader holds a time-bounded guarantee — "no new leader can be elected for the next T milliseconds." It can serve reads locally without a quorum check. Fast, but only safe if clocks don't drift past the lease duration. One clock skew incident and you've got a split-brain read.

Read index (Raft): before serving a read, the leader records its current commit index, sends a heartbeat round to confirm it's still the leader, waits until its applied index catches up to the commit index, then serves the read. No clock dependency, strictly safe — costs a tiny bit of latency.

Distributed locks & fencing tokens
Safe mutual exclusion across nodes
Practical

Here's the failure mode: a process holds a lock, gets paused — GC, a slow network call, anything — and resumes after the lock expired and was re-granted to someone else. It has no idea its lock is stale and tries to write.

Fencing tokens: every time a lock is granted, issue a monotonically increasing token. The storage system tracks the highest token it's seen and rejects writes from anything lower.

TTL alone is insufficient — a paused client's internal clock doesn't advance during the pause. It wakes up thinking barely any time passed. The storage system must enforce fencing, not just the lock service.

Redlock controversy: Redis's multi-node Redlock is a classic interview talking point. Martin Kleppmann's critique: it relies on bounded clocks and process pauses, and still needs fencing tokens to be safe — so for correctness-critical locking, prefer a consensus system (ZooKeeper/etcd) that issues fencing tokens natively.

FLP impossibility
Fundamental limit of consensus
Theory

The FLP result (Fischer, Lynch, Paterson 1985): in a purely asynchronous system — where messages can be delayed arbitrarily — you can never distinguish a crashed node from one that's just slow. So no deterministic consensus algorithm can guarantee it always terminates, even if only one process fails.

Real systems work around this by relaxing the pure asynchrony assumption. Raft and Paxos both use timeouts — they assume the network is eventually synchronous, not always asynchronous. That's called partial synchrony, and it's enough to make consensus practical. Randomised algorithms are another escape: they terminate with high probability even under full asynchrony.

Byzantine fault tolerance
When nodes can lie, not just crash
Theory

Raft and Paxos handle crash faults — nodes that stop responding. Byzantine faults are worse: a node can behave arbitrarily, send conflicting messages to different peers, lie, or be actively compromised. The question is whether honest nodes can still agree.

The 3f+1 bound: to tolerate f Byzantine nodes you need 3f+1 total — versus 2f+1 for crash faults. The intuition: faulty nodes can actively lie, so you need a large enough honest majority that even after some honest nodes are unreachable, the remaining honest nodes still outvote the liars.

PBFT made this workable with a three-phase protocol (pre-prepare → prepare → commit). Blockchains go further — Nakamoto consensus (proof-of-work) and modern BFT protocols like Tendermint and HotStuff tolerate Byzantine actors at internet scale by making the cost of lying prohibitively high.

Interview reality: inside a single trusted datacenter you almost never need BFT — crash-fault consensus is enough and far cheaper. Reach for BFT only across trust boundaries (blockchains, multi-org systems).

Distributed transactions

Achieving ACID properties across multiple nodes or services. One of the hardest problems — most modern systems opt for weaker guarantees in exchange for availability and performance.

ACID properties
What a transaction actually guarantees
Foundational

ACID is the bundle of guarantees a transactional database gives you. Worth knowing what each letter actually means — and where each becomes hard in a distributed system.

A — Atomicity: all-or-nothing. Every operation in the transaction commits, or none of them do. If the system crashes mid-transaction, the DB rolls back to where you started. Implemented via write-ahead logging — changes go to the log before being applied, so recovery can undo incomplete work.

C — Consistency: the transaction takes the DB from one valid state to another, never breaking constraints like foreign keys, uniqueness, or check constraints. One nuance: this is an application-level invariant enforced by the DB, not a distributed systems property.

I — Isolation: concurrent transactions behave as if they ran one at a time — no transaction sees another's intermediate state. Full serializability is expensive, so most systems run at a weaker level like Read Committed or Snapshot Isolation and accept certain anomalies. See the isolation levels card for the specifics.

D — Durability: once you commit, it survives crashes. The DB flushes the WAL to disk before confirming the commit. Replication adds another layer: with semi-synchronous replication, at least one follower must persist the write before the ack goes out — so you're safe even if the leader's disk fails immediately after committing.

Don't confuse the two "atomicity"s: the A in ACID (a transaction is all-or-nothing) is completely unrelated to atomic/linearizable consistency (single-copy real-time ordering). These share a word but describe entirely different properties. Interviewers probe this conflation deliberately.
ACID in distributed systems: Atomicity across nodes requires a commit protocol (2PC). Isolation across nodes requires distributed locking or MVCC coordination (Spanner, CockroachDB). Durability across nodes is achieved via replication. Achieving all four simultaneously is possible (Spanner does it) but expensive — most microservice architectures relax ACID and use Sagas or eventual consistency instead.

Two-phase commit (2PC)
Atomic commit across multiple nodes
Classic

Phase 1 — Prepare: the coordinator asks every participant "can you commit?" Each participant durably logs its vote and responds yes or no. The critical detail: once a participant votes yes, it's locked in — it cannot unilaterally abort, it has to wait for the coordinator's decision.

Phase 2 — Commit: if all participants voted yes, the coordinator sends Commit and everyone applies the transaction. Any no vote → the coordinator sends Abort.

Normal flow — unanimous commit
Unanimous YES → the coordinator broadcasts Commit and everyone applies. One NO (or a timeout before deciding) → it broadcasts Abort instead.
Coordinator crashes after Prepare, before Commit
The coordinator dies after collecting YES votes but before broadcasting the decision. Both participants are in-doubt — they can't commit (maybe it aborted) or abort (maybe it committed), so they block, holding locks, until it recovers. That's the coordinator SPOF and why 2PC is a blocking protocol.

The blocking problem — this is the key weakness: if the coordinator crashes after sending Prepare but before sending the Commit or Abort decision, participants are stuck indefinitely. They voted yes, they don't know the outcome, and they're holding locks. Other transactions pile up behind them. This is why 2PC is called a blocking protocol — a coordinator failure can halt progress until it recovers.

3PC adds a pre-commit phase to make this non-blocking, but it's rarely used in practice — it breaks under network partitions and adds a round trip. The modern answer is to make the coordinator itself fault-tolerant via consensus, like Spanner running 2PC over Paxos groups.

Saga pattern
Long-running transactions via compensation
Microservices

Instead of one distributed transaction, you break the work into a sequence of local transactions, each publishing an event when it succeeds. If a step fails, you run compensating transactions in reverse to undo what already completed.

Forward (happy path) — a step fails
Each step is a local txn that commits and emits an event. Order ✓ and Payment ✓ already committed — then Inventory fails. There is no global rollback to undo the first two.
Compensation — run in reverse
Run compensating txns in reverse: C2 refunds the charge, C1 cancels the order. Each C is a semantic undo (a refund/cancel row), NOT a DB rollback — and must be idempotent so retries are safe.

Choreography: each service reacts to events from others — decentralised, hard to track overall progress.

Orchestration: a central saga orchestrator sends commands to each service — clearer flow, easier to debug, single point of control.

Critical requirements: every step must be idempotent (safe to retry), and every step must be compensable (semantic undo is possible). There's no atomic rollback — compensation is an application-level undo that may itself fail.

Idempotency & at-least-once delivery
Safe retry semantics
Critical

Networks duplicate messages. Clients retry on timeout not knowing if the first attempt succeeded. Design operations to be idempotent: applying the same operation multiple times has the same effect as once.

Idempotency keys: client generates a UUID per request. Server stores the key and result. On retry, returns the stored result without re-executing.

Natural idempotency: PUT (set to this value) is naturally idempotent; POST (create) is not.

Conditional writes: write only if current version matches — rejects duplicate writes that would update an already-updated record.

Exactly-once effect/processing is the practical goal — true exactly-once delivery over a network is impossible (two-generals), so you get there with at-least-once delivery + idempotent consumers, which yields the same observable result far more cheaply.

TCC (Try-Confirm-Cancel)
Reservation-based distributed transactions
Pattern

A business-level alternative to 2PC and a more structured saga. Each participant exposes three operations:

Try: reserve resources without committing (e.g. hold inventory, freeze funds). Confirm: commit using the reserved resources — must be idempotent and must not fail. Cancel: release the reservation.

Try → reserve 1 seat, hold $50 Confirm → convert hold into a real booking + charge Cancel → release seat, unfreeze $50

Unlike 2PC, locks are application-level reservations (not DB row locks), so resources aren't held by a blocking protocol. Unlike a raw saga, the "Try" phase verifies feasibility up front, reducing the chance of needing compensation. The cost: every service must implement all three phases and a reservation TTL to auto-cancel abandoned tries.

Snapshot Isolation (MVCC) is what most databases actually implement when they say "Repeatable Read." Concurrency control mechanisms
How databases actually enforce isolation
Internals

Two-phase locking (2PL): a growing phase acquires locks, a shrinking phase releases them at commit. Shared (read) locks coexist; exclusive (write) locks don't. Gives true serializability but throttles concurrency and invites deadlocks. Predicate / index-range locks extend it to block phantoms.

Deadlocks: a cycle of transactions each waiting on a lock the other holds. Handle by detection (build a waits-for graph, abort a victim) or prevention (consistent lock ordering, lock timeouts, or wait-die / wound-wait schemes).

Optimistic concurrency control (OCC): take no locks; at commit, check whether anything you read was modified meanwhile — if so, abort and retry. Wins when contention is low; wasteful when it's high.

MVCC: keep multiple versions of each row so readers see a consistent snapshot and never block writers (and vice versa). It's the engine behind Snapshot Isolation. SSI layers conflict detection on top to reach serializable while staying mostly lock-free — what PostgreSQL's Serializable does.

Pessimistic vs optimistic: pessimistic (2PL) assumes conflict and locks up front; optimistic (OCC/SSI) assumes conflict is rare and validates at commit. Choose by your expected contention.

Partitioning & sharding

Splitting data across nodes to distribute load and enable horizontal scaling. The strategy determines query efficiency, hot-spot risk, and rebalancing cost.

Range partitioning
Contiguous key ranges per partition

Assign contiguous key ranges to partitions (A–F on node 1, G–M on node 2, etc.). Efficient for range queries and sorted scans — all keys in a range live on one node.

Hot partition risk: if timestamps are used as keys, all current writes hit the same partition. Solution: prefix keys with a random hash bucket (e.g. hash(user_id) % 8 + "::" + timestamp) to spread writes while keeping per-user records sortable.

Hash partitioning & consistent hashing

Uniform load

Hash the partition key, node = hash(key) % N. Distributes load evenly but destroys sort order — range queries require scatter-gather across all nodes.

Consistent hashing: place nodes and keys on a virtual ring. A key goes to the first node clockwise from its hash position. Adding or removing a node only rebalances ~1/N of keys rather than everything.

Rebalancing strategies
Moving data when nodes are added or removed
Operations

Fixed partitions: create many more partitions than nodes (e.g. 1000 partitions for 10 nodes). Adding a node moves whole partitions — no data splitting needed. Simple and predictable. Used by Elasticsearch, Riak.

Dynamic partitioning: partitions split when they exceed a size threshold and merge when they shrink. Partition count grows with data volume. Used by HBase, MongoDB.

Proportional partitioning (Cassandra): fixed number of vnodes per physical node. Adding a physical node splits existing vnodes proportionally — keeps partition sizes stable as nodes are added.

Avoid hash mod N for rebalancing: never tie partition count to node count. Decouple them (fixed partitions or consistent hashing) so adding a node doesn't reshuffle the whole dataset.
Request routing & service discovery
"Which node has my key?"
Topology

Once data is partitioned, a client needs to find the right node — and the mapping changes as nodes join/leave. Three approaches:

1. Routing tier / proxy: a dedicated layer (e.g. mongos, Twemproxy, Vitess) knows the partition map and forwards requests. Clients stay dumb.

2. Client-side routing: a partition-aware client fetches the topology and routes directly (Cassandra, DynamoDB SDKs) — one fewer hop, but the client must track membership.

3. Any-node redirect: send to any node; if it doesn't own the key it forwards or redirects (Cassandra gossip, Redis Cluster MOVED).

Time, ordering & failure detection

In a distributed system, clocks drift, messages are delayed, and you can never be certain whether a node has crashed or is just slow. These primitives are foundational.

Lamport clocks
Logical time for causal ordering
Foundational

Simple rules: each process maintains a counter. On any local event: increment. On send: attach the counter to the message. On receive: set your counter to max(local, received) + 1.

This gives you a useful partial order: if A causally happened before B, then Lamport(A) < Lamport(B). But the converse isn't true — a lower clock value doesn't mean the event came first causally. Lamport clocks can't distinguish causally ordered events from concurrent ones. For that, you need vector clocks.

Lamport 1978partial order onlytotal order possible with tie-breaking
Vector clocks
Precise causal relationships
Causality

Each process maintains a vector — one counter per process. On send: increment your own slot in the vector and attach the full vector to the message. On receive: take the element-wise max of your vector and the received one, then increment your own slot.

V(A) < V(B) iff every entry of A ≤ B's, and at least one strictly less → A causally happened-before B Neither V(A) ≤ V(B) nor V(B) ≤ V(A) → A and B are concurrent — no causal relationship

This gives you the full happened-before relation — you can now tell the difference between "A caused B" and "A and B happened independently." Dynamo and Riak use this for conflict detection. The practical downside: vector size grows with the number of processes. In practice, systems use version vectors — the same per-replica counters, but scoped to a single key/object rather than the whole process history — so the vector only grows with the number of replicas, not the global event count.

TrueTime (Google Spanner)
Bounded uncertainty with GPS + atomic clocks
Novel

Instead of returning a point in time, TrueTime returns an interval [earliest, latest] — a guarantee that the true current time falls somewhere in that window. GPS receivers and atomic clocks in every Google datacenter keep that uncertainty (ε) to single-digit milliseconds, typically under 4ms.

Spanner uses this for external consistency: before releasing a transaction's commit timestamp, it waits out the full uncertainty window. The cost is latency proportional to ε, which is why Google invests heavily in the hardware to keep ε small.

The key insight: rather than pretend clocks are perfect, TrueTime makes the uncertainty explicit and then waits it out. Smaller ε = shorter commit wait = lower latency, which is why Google invests in atomic clocks. HLC is the "no special hardware" alternative.

Distributed trouble

Networks drop packets, nodes pause, and there is no global clock — so you can never be sure a peer is dead, only unresponsive. How distributed systems detect failure and spread state despite partial failures.

Failure detection
Timeouts, gossip, Phi accrual
Operations

The fundamental problem: in an asynchronous network you can never perfectly distinguish a crashed node from one that's just slow. FLP in practice — all failure detectors are making a judgment call, not stating a certainty.

Heartbeat + timeout: the simple approach. If no heartbeat arrives within the timeout, assume dead. Hard to tune: too short a timeout and you get false positives and unnecessary failovers; too long and failure detection is slow.

Gossip protocol: nodes randomly swap membership state with a few peers each round. Failure information propagates in O(log N) rounds across the cluster. The Dynamo paper, Cassandra, and Riak use this — it's decentralised and keeps working even as nodes fail. (AWS's DynamoDB product is a different internal design — don't conflate it with the Dynamo paper.)

Φ accrual detector (Cassandra): instead of a binary alive/dead verdict, it outputs a continuous suspicion value Φ that you threshold yourself based on your false-positive tolerance. It adapts to actual network conditions rather than using a fixed timeout — so it gets tighter when the network is healthy and more lenient when latency spikes.

gossipphi accrualpartial synchrony
Gossip protocol
Epidemic dissemination without a coordinator
Membership

How do thousands of nodes agree on "who's alive" with no central registry? Each node periodically picks a few random peers and exchanges state. Information spreads like a rumour (or a virus) — every round the number of informed nodes roughly multiplies, so the whole cluster converges in O(log N) rounds.

Two flavours:

Anti-entropy: peers compare their full state and reconcile differences (often via Merkle trees so they only ship the diff). Continuous and thorough — guarantees eventual convergence even for state nobody is actively touching.

Rumour-mongering (dissemination): when something changes, gossip just that update for a while, then stop. Fast and low-bandwidth for propagating fresh events.

round 0: 1 node knows round 1: ~2 know round 2: ~4 know … → O(log N) rounds to reach all N

Why it's used: no single point of failure, no coordinator to overload, and it degrades gracefully as nodes come and go. It carries membership + failure suspicion (often feeding a Φ-accrual detector) and metadata. Used in Cassandra, Riak, Consul/Serf (SWIM variant), and Redis Cluster.

Tradeoff: eventually consistent membership — there's propagation delay, so two nodes can briefly disagree on who's up. Fine for membership/discovery; not a substitute for consensus when you need a single authoritative decision (use Raft/ZooKeeper for that).

Delivery semantics
At-most / at-least / exactly-once
Critical

At-most-once: fire and forget. Messages may be lost, never duplicated. Fine for metrics/telemetry where loss is acceptable.

Backpressure & DLQs
Flow control and poison messages
Failure modes

Backpressure: when consumers can't keep up, the system needs to slow producers rather than crash. Pull-based systems like Kafka get this for free — consumers fetch at their own pace, lag builds up but nothing falls over. Push-based systems need explicit credit or window protocols (Reactive Streams, gRPC flow control), otherwise the consumer gets overwhelmed.

Bounded buffers: never use unbounded queues — they just move the failure to OOM later. Bound the buffer and decide a policy when it's full: block the producer, drop the oldest messages, or shed load.

Poison messages and DLQs: a message that always fails will block a partition or loop forever. After N retries (with backoff), route it to a dead-letter queue for human inspection instead of letting it block the pipeline. Pair with retry topics for delayed reprocessing when you want to try again later.

Caching

Reduces latency and backend load. Key challenges: cache invalidation, thundering herd, and maintaining acceptable consistency with the source of truth.

Cache-aside (lazy loading)
Application manages the cache explicitly
Most common
read(key): val = cache.get(key) if val is None: # cache miss val = db.get(key) # 2 round trips on miss cache.set(key, val, ttl=300) return val

On a cache miss you pay two round trips — one to the DB, one to populate the cache. Data can go stale between population and TTL expiry. Works well for read-heavy workloads where most data is accessed infrequently. The TTL is your only consistency mechanism — too short and you lose the benefit, too long and you serve stale data.

Invalidation, not update: on a write, delete the cache key rather than updating it. Concurrent update-then-set races can leave a stale value cached forever; delete-on-write lets the next read repopulate cleanly.
The populate-on-miss race: a reader misses and fetches the old DB value; before it writes that value back, a writer updates the DB and deletes the key — so the reader repopulates a stale value that lives until TTL. Cheap defense: a short TTL + single-flight. Full defense: a versioned (compare-and-swap) set that rejects a write whose version is older than what's already cached. This race is unique to cache-aside — write-through and write-back populate on the write path, so they never read-then-populate.

Read-through (cache loads on miss)
Same lazy-load effect — but the cache does the loading, not the app
Read path

Read-through produces the same effect as cache-aside — on a miss, data is loaded from the DB into the cache — but it differs on who orchestrates the load. In cache-aside the application reads the DB and populates the cache. In read-through the cache itself has a loader and fetches from the DB on a miss, so the app only ever talks to the cache.

The red edge is the whole distinction: it moves from App→DB to Cache→DB. Read-through keeps app code simple (no miss-logic duplicated at every call site), but it needs a cache that supports a loader and couples the cache to your data source. Cache-aside is the common default because plain Redis/Memcached have no loader, and the app keeps working if the cache is down (it just hits the DB).

Pairs with write-through: read-through is usually combined with write-through — the same idea on the write path, where the cache (not the app) propagates to the DB. Examples: AWS DAX in front of DynamoDB; Caffeine/Ehcache CacheLoader.

Cache stampede (thundering herd)

Failure mode

When a popular cached key expires, thousands of concurrent requests miss the cache simultaneously and all hit the database — potentially crashing it before the cache is repopulated.

No guard

Singleflight/mutex collapses the herd. One request rebuilds; the rest wait or serve stale. Jittered TTLs and stale-while-revalidate reduce the chance of the herd forming.

Solutions:

Mutex lock: only the first request to miss fetches from DB; all others wait on a lock. Adds latency for waiters, but prevents DB overload.

Probabilistic early expiration (PER): before the TTL expires, randomly decide to refresh early based on remaining TTL and fetch cost. Smoothly pre-warms the cache.

Stale-while-revalidate: serve the stale cached value immediately while a background job refreshes it. No user-visible latency, slightly stale data.

Cache warming (pre-warming): proactively load the hot keys before traffic hits them — on deploy/restart/failover, or ahead of a predictable spike (flash sale, launch). This is the one fix for a cold cache: single-flight has nothing to coalesce and stale-while-revalidate has no stale value to serve when the whole cache is empty, so warming the hot set is what prevents the cold-start herd. Refresh-ahead (background-refresh hot keys before TTL) is the steady-state version.

Bloom filters

Probabilistic

A Bloom filter is a probabilistic set membership structure — a bit array with k hash functions. It only gives you two answers: "definitely not present" or "maybe present." No false negatives, but tunable false positives. You can't delete from it (use a counting Bloom filter if you need deletes).

add(x): set bits at h1(x), h2(x), ... hk(x) = 1 has(x): all those bits = 1? → maybe | any bit = 0? → definitely not
A 16-bit array, k = 3 hash functions. Add items to set their bits to 1, then query membership. Add all three items first, then query cow — it was never added, yet all its bits happen to be set.

Health checks & failover
Routing around broken backends
Availability

Active checks: LB probes /healthz periodically; unhealthy backends are removed from rotation. Passive checks: infer health from real traffic (e.g. consecutive 5xx / connection failures eject the host — "outlier detection" in Envoy).

Shallow vs deep: a shallow check ("am I up?") can pass while dependencies are down; a deep check (DB reachable, etc.) catches more but risks cascading removal if a shared dependency blips — suddenly every backend fails its check at once.

The health-check death spiral: aggressive deep checks + a shared dependency hiccup can eject all backends at once, taking the whole service down. The fix worth naming: fail open — when "too many" hosts look unhealthy, assume the check is wrong and keep serving.

For high availability the LB itself must be redundant (active-passive with a floating VIP, or anycast) — it's otherwise a single point of failure.

Topology

Geo / global server load balancing (GSLB): direct users to the nearest or healthiest region. Two main mechanisms: DNS-based (return region-specific IPs; simple but DNS caching slows failover) and anycast (the same IP announced from many PoPs; the network routes to the closest — fast failover, used by CDNs and DNS providers).

Sticky sessions (affinity): pin a client to one backend (via cookie or hash) so in-memory session state stays valid. Convenient, but it breaks even load distribution and loses state on failover.

Prefer stateless: the cleaner answer is to externalize session state (Redis, signed JWTs) so any backend can serve any request — then you don't need stickiness and scaling/failover are trivial. Mention this whenever sticky sessions come up.

Rate limiting & throttling

Protecting services from overload, abuse, and noisy neighbours. The algorithm determines burst tolerance and smoothness; doing it across a fleet adds a coordination problem.

Token bucket
Allows controlled bursts
Most common

A bucket holds up to B tokens and refills at R tokens/sec. Each request consumes a token; if the bucket is empty, the request is rejected (or queued). Allows bursts up to the bucket size while bounding the long-run average rate.

Stored compactly as (token_count, last_refill_time) — compute refill lazily on each request. The de-facto default (AWS, Stripe, NGINX). Tune B for burst tolerance and R for sustained rate independently.

burst-friendlytwo params: B, Rlazy refill
Leaky bucket
Smooths output to a constant rate
Smoothing

Requests enter a fixed-size FIFO queue and are processed ("leak") at a constant rate. Overflow is dropped. Unlike token bucket, it does not allow bursts to pass through — it enforces a steady outflow, which is ideal for protecting a downstream that needs smooth traffic.

Token bucket vs leaky bucket: token bucket limits the average but lets bursts through; leaky bucket shapes traffic into a constant stream. Choose by whether the protected resource tolerates bursts.

Atomicity & failure
The race, and when the limiter is down
Gotcha

Make the check atomic. Even with a shared Redis counter, a naive read-modify-write races: two requests both read "1 token left", both allow, both spend it — you served 2 on a limit of 1. Fail open vs fail closed. The limiter fronts every request, so decide up front what happens when Redis is unreachable. Fail closed (reject, or fall back to a conservative local limit) protects the backend from a cascade at the cost of some legitimate traffic. For abuse / overload protection, fail closed is usually right.

Back-of-envelope estimation
QPS, storage, bandwidth
Method

A repeatable recipe interviewers look for:

1. Traffic: DAU × actions/user/day → requests/day → ÷ 86,400 for average QPS. Peak ≈ 2–10× average. Split read:write (often 100:1 or more for social/feed apps).

2. Storage: writes/day × bytes/write × retention. Add replication factor (×3) and indexes/overhead. Project growth over years.

3. Bandwidth: QPS × payload size, in and out separately.

Vertical vs horizontal scaling
Bigger box vs more boxes
Tradeoffs

Vertical scaling (scale up): get a bigger machine. Simple, no application changes, keeps consistency easy. But there's a hard ceiling on how big a single machine can get, costs grow non-linearly, and you're still a single point of failure.

Horizontal scaling (scale out): add more machines. Effectively unbounded and fault-tolerant — but now you have to deal with partitioning, replication, consistency, and coordination. Which is what everything else in this guide is about.

The key enabler is statelessness: push all state out of app servers and into databases, caches, and object stores. Then any instance can handle any request, and horizontal scaling becomes as simple as "add more instances behind a load balancer." The database becomes the new bottleneck — which is why read replicas, sharding, and caching exist.

Amdahl & Universal Scalability Law: Amdahl tells you speedup is always capped by the serial fraction of your workload. The USL goes further — it adds a coordination/contention penalty that can actually make throughput decrease past a point if nodes spend too much time cross-talking. More machines doesn't always mean more throughput.

Reliability & resilience

Failure is the normal case at scale. These patterns keep one failing dependency from taking down the whole system, and these metrics define what "reliable enough" means.

SLI, SLO, SLA & error budgets
Defining and buying reliability
Foundational

SLI (Indicator) — a measured number: e.g. % of requests <200ms, success rate. SLO (Objective) — the internal target for an SLI: e.g. 99.9% success over 30 days. SLA (Agreement) — a contractual promise to customers, with penalties; always looser than the SLO.

Error budget: 100% − SLO. A 99.9% SLO means a 0.1% budget ≈ 43 min/month of allowed unavailability. This reframes reliability as a resource you spend: budget left → ship features faster; budget exhausted → freeze releases and fix stability.

Circuit breaker
Stop hammering a failing dependency
Pattern

Wrap calls to a remote dependency in a breaker with three states. When failures cross a threshold, the breaker opens and fails fast immediately — instead of waiting on timeouts — giving the dependency room to recover and freeing the caller's threads.

This prevents the classic cascading failure: a slow dependency ties up every caller thread, the caller's queue fills, and the outage spreads upstream. Always pair circuit breakers with sensible timeouts — a call with no timeout is the root cause of most cascading failures — and define a fallback for the open state.

Retries, backoff & jitter
Retrying without making it worse
Practical

Retries recover from transient faults — but naive retries amplify an overload. Three rules:

1. Exponential backoff: wait 1s, 2s, 4s, 8s… so you back off as the dependency struggles, instead of pounding it.

2. Jitter: add randomness to each delay. Without it, all clients that failed together retry together — a synchronized retry storm / thundering herd. sleep = random(0, base * 2^attempt) (AWS "full jitter").

3. Only retry idempotent/retryable errors, cap attempts, and use a retry budget (e.g. retries ≤ 10% of requests) so retries can't more than fractionally add to load.

No jitter
Full jitter
Backoff without jitter synchronizes clients. Full jitter picks a random delay inside the backoff window, spreading retries across time.
Retry amplification: if every layer retries 3×, a 3-layer call stack can multiply one user request into 27 backend calls during an incident. Retry at one layer, fail fast at the others.

Bulkheads & load shedding
Isolation and graceful degradation
Containment

Bulkheads (named after ship hull compartments): give each downstream dependency its own thread pool or connection pool. If one saturates, the others keep serving. One slow call type can't starve everything else.

Load shedding: when you're overloaded, deliberately reject low-priority work early — return 429 or 503 — to protect your ability to serve the rest. It's better to serve 90% of requests well than to fall over trying to serve 100% badly.

Graceful degradation: drop to a reduced but functional mode instead of failing the whole request — serve stale cache, hide the recommendations widget, disable expensive features. The key is defining what the "lite" mode looks like in advance, not figuring it out during an incident.

Graceful shutdown & connection draining
Don't drop in-flight work on deploys
Operations

On a rolling deploy or scale-down, a node is told to stop — but it may still be serving requests. Killing it immediately drops those in-flight requests, so users see 5xx errors and connection resets during routine rollouts.

Connection draining: first deregister the node from the load balancer (or fail its readiness check) so no new requests are routed to it. Then let the in-flight requests finish on their own, up to a grace timeout, before the process exits.

The shutdown sequence: on SIGTERM — stop accepting new connections, finish (or hand off) existing ones, flush buffers/commit offsets, then exit. Long-lived streams (WebSockets, gRPC) should stop accepting new streams and let active ones complete or reconnect elsewhere.

Interview tip: pair draining with health checks — the LB should see the node as unhealthy before it stops, and the grace period must be longer than your slowest normal request. This is how zero-downtime deploys actually work.

Multi-region strategy & DR
Surviving a whole-region outage
Operations

Running in multiple regions buys lower latency for global users and survival of a full-region failure — at the cost of cross-region replication and conflict handling.

Active-passive (failover): one region serves; a standby replicates and takes over on failure. Simpler and consistent, but the standby sits mostly idle and failover takes time.

Active-active: all regions serve traffic, routed by geo/latency. Best latency and utilization, but concurrent writes in different regions force a conflict-resolution strategy (LWW, CRDTs, or pin each key to a home region).

RPO vs RTO: RPO (Recovery Point Objective) = how much data you can afford to lose — bounded by replication lag (sync ≈ 0, async ≈ the lag). RTO (Recovery Time Objective) = how fast you must be back — bounded by failover automation and DNS/anycast propagation.

RPO ≈ replication lag (sync → 0, async → lag) RTO ≈ detect + promote + reroute time

Also consider: data residency / sovereignty (GDPR — some data can't leave a region), failover testing (an untested failover doesn't work), and split-brain avoidance (fenced/consensus-based promotion, never two primaries).

Schema migrations & backfills
Changing data with zero downtime
Operations

You can't take the DB down to change a column, and old + new code run simultaneously during a deploy. The safe pattern is expand-contract (parallel change):

1. Expand: add the new column/table — additive, backward-compatible. Old code ignores it.

2. Migrate: deploy code that dual-writes (old + new), then backfill historical rows in batches.

3. Contract: switch reads to the new field, stop writing the old one, then drop it — only once nothing reads it.

expand → dual-write + backfill → switch reads → contract (drop old) each step independently deployable & rollback-safe

Backfill safety: run in rate-limited batches (don't saturate the DB or replication), make it idempotent and resumable (checkpoint progress), and verify counts before contracting. Never rename-in-place or drop-then-add in one shot — it breaks the old code still running mid-deploy.

Security & abuse

The cross-cutting concerns interviewers probe once the happy path works: who is allowed to do what, whether reads/writes are scoped to the right data, how bad traffic is handled, and how secrets/PII stay protected. (Token mechanics live in Data models & encoding; volume controls in Rate limiting.)

Authorization & data access
Who you are, what you may do, and which rows you may touch
Foundational

Authentication (AuthN) — who are you? Verify identity with a password, token, or mTLS certificate. Happens once, at the edge. Authorization (AuthZ) — what may you do? Check the caller's rights against the specific action, on every request.

Authorization models: RBAC — permissions via roles (admin, editor); simple but coarse, and roles explode as rules get fine-grained. ABAC — rules over attributes (dept=HR AND region=EU); flexible but harder to audit. ReBAC — permissions from relationships ("owner of doc X"); Google's Zanzibar-style model, the right fit for sharing graphs.

Data access isolation: every read/write must be scoped to the authorized user, org, project, workspace, or account from the verified token — never from a request parameter. The classic bug is changing /orders/123 to /orders/124 and seeing someone else's data.

Defense in depth: centralize authorization checks, use database constraints or row-level security where appropriate, and make service-to-service calls authenticate too with mTLS or short-lived signed tokens. "It's on the private network" is not authentication.

Cardinal rule: authenticate once, authorize every time. A valid token is identity, not permission — a logged-in user can still be unauthorized for the specific action.

Abuse & input defense
Injection, bots, credential stuffing, and bad traffic
Defensive

Bad input: validate input and encode output — the defense against injection (SQLi, XSS). Parameterize queries; never string-concatenate user input.

Bad actors: defend against bots and credential stuffing with rate-limit + lockout, CAPTCHA or challenge flows, device fingerprinting, and anomaly detection. A WAF can block known-bad patterns at the edge.

Authorized abuse: a logged-in caller can still be abusive. Put authorization on every action, quotas around expensive work, and fraud/anomaly signals around slow, valid-looking attacks.

Rate limits aren't enough: they stop obvious volume spikes, but not low-and-slow credential stuffing, scripted account takeover, or a single authorized account draining expensive resources.

Security & privacy basics
TLS, encryption, secrets, audit logs, and PII retention
Practical

Data protection: use TLS in transit and encryption at rest. For sensitive fields, tokenize or field-encrypt PII so a leaked DB dump is less useful.

Secrets: no secrets in code, git, container images, or logs. Use a secrets manager or KMS-backed vault, inject at runtime, and rotate keys or credentials when needed.

Audit & privacy: log sensitive/admin actions in an append-only audit log. Minimize PII collection, scrub PII/secrets from logs, and define retention/delete behavior when the domain needs it.

Handling large blobs
Video, images, big files
Pattern

Never route gigabytes through your app servers. Instead, the server issues a temporary, scoped presigned URL and the client uploads directly to blob storage (S3). Downloads go through a CDN with signed URLs for access control.

This removes your servers as the bottleneck and enables resumable uploads and progress tracking. Main challenge: keeping DB metadata in sync with storage — use storage event notifications to confirm uploads and reconcile state.

Chunk on the client. Split big files into 5–10 MB chunks on the device before uploading: each chunk uploads independently, so a dropped connection resumes from the last committed chunk instead of byte zero, and you dodge the ~10 MB request-body cap most API gateways impose. Chunking server-side defeats the purpose — the client already had to send the whole file once.

Content-defined chunking & dedup. Split on a rolling-hash boundary rather than fixed offsets, so a small edit re-uploads only the few chunks around the change instead of shifting every boundary after it. Hash each chunk (e.g. SHA-256) to a content fingerprint: skip chunks already present (resumable) and share identical bytes across users (cross-user dedup) — the fingerprint identifies content, not a record, so two users still get distinct file IDs.

Trust but verify. Don't mark an upload complete because the client said so — confirm the parts actually landed via the multipart ListParts / ETags first. And if you compress to save bandwidth, compress before encrypting (ciphertext won't compress) and skip it for already-compressed media.

Pattern

When one user posts, that event may need to reach millions of followers' feeds or inboxes. The core challenge is fan-out — write amplification when a single user has millions of followers. The design choice is where you pay the cost: on write or on read.

Fan-out on write (push model): when a user posts, immediately write a feed/inbox entry to every follower's timeline. Fast reads (inbox is pre-populated). Prohibitive for celebrities — a user with 10M followers triggers 10M writes on a single post.

Fan-out on read (pull model): store the event once; followers compute their feed/inbox at read time. Cheap writes, expensive reads — and read latency grows with the number of people you follow.

Hybrid (the standard answer): fan-out on write for normal users (< some follower threshold); fan-out on read for high-follower "celebrity" accounts. Readers merge their pre-populated inbox with any celebrity events at read time.

Proximity-based services
"Find things near me"
Pattern

Location search (Uber, Yelp, food delivery) needs a geospatial index. Divide the map into regions so the system can exclude vast empty areas and shrink the search space. Implementations: PostgreSQL + PostGIS, Redis geospatial type, or Elasticsearch geo-queries; algorithms like geohash, quadtrees, S2/H3 cells, or R-trees (the bounding-box index behind PostGIS).

The trick (geohash): encode (lat, lng) into a short string where a shared prefix means physical closeness — 9q8yy and 9q8yz are adjacent cells, 9q8 is the bigger block containing both. That collapses 2D proximity into a 1D prefix range, so a plain B-tree or a Redis sorted set can answer "near me" with an ordinary range scan — no special index engine required. Longer prefix = smaller, more precise cell.

Design archetypes

Most interview prompts are one of a handful of recurring problem families. Recognise the archetype and its playbook — the core decisions, the canonical design, the classic trap — and you've already framed the problem before drawing a box. The Common patterns section is the mechanism layer (fan-out, geo, blobs); this is the layer above it: whole-problem families that those mechanisms compose into. And they stack — a video platform is blob delivery + async pipelines + real-time. Below: the eight families, then drills to train the prompt → archetype reflex.

The interview framework

Concepts win you points; structure wins you the interview. A repeatable approach keeps you from freezing on an open-ended prompt and shows the interviewer how you think.

A delivery framework
A repeatable ~45-minute structure
Method

1. Requirements (~5m): functional ("users should be able to…") — prioritize the top 3; a long list hurts you. Then non-functional, quantified: "feed <200ms", "100M DAU", "availability over consistency". Pin scope explicitly — you can't design everything.

2. Core entities (~2m): jot the key nouns/resources (User, Tweet, Follow) before schemas — flesh out fields later as the design reveals them.

3. API / interface (~5m): define the contract. Default to REST with plural resource nouns; GraphQL for diverse clients; gRPC for internal perf. Derive the current user from the auth token, never from the request body.

4. [Optional] Data flow (~5m): for data-processing systems, list the pipeline (fetch → parse → extract → store).

5. High-level design (~10–15m): draw boxes that satisfy each API endpoint one by one — clients, LB, services, DB, cache, queue. Walk one request end to end; resist adding complexity early — note it and move on.

6. Deep dives (~15m): the interviewer steers here. Harden against the non-functional requirements: go deep on 1–2 components (sharding scheme, consistency model, the hot path, a failure mode), edge cases, and bottlenecks. Your seniority shows in how proactively you lead this.

Estimation — inline, not upfront: skip ritual back-of-envelope math; do it only when a number will actually change the design (e.g. "does the Top-K heap fit on one node?").

Pitfalls

Designing before clarifying. Jumping to a diagram without nailing requirements signals you don't probe ambiguity.

Over-engineering. Reaching for Kafka, microservices, and global multi-region when the prompt is 1K users. Start simple, scale when asked.

Hand-waving the hard part. Saying "and then we shard" without explaining the key, hot-spots, or rebalancing. The deep dive is where you're actually scored.

Data models & wire encoding
Shape the data, then serialize it
Foundational

Pick a data model by relationship shape: Relational (tables + joins) for many-to-many and ad-hoc queries; Document (nested JSON per entity) for one-to-many with locality and a flexible schema; Graph (nodes + edges) for many-to-many with multi-hop traversals. (Which store implements it is a Design decisions question.)

Wire formats: JSON / XML are human-readable and universal but verbose and weakly typed; Protobuf / Avro / Thrift carry a schema and encode to compact, fast binary — preferred for internal service-to-service and high-volume pipelines.

Schema evolution: producers and consumers deploy independently, so changes must be backward-compatible (new code reads old data) and forward-compatible (old code reads new data). The rule depends on the format: Protobuf / Thrift identify fields by integer tags — add only optional fields and never reuse a tag number. Avro has no tags; it matches fields by name using the writer's + reader's schemas, with defaults filling any gaps (so a removable/addable field needs a default). A schema registry (e.g. Confluent) is what lets Avro producers and consumers fetch each other's schemas and evolve independently.

Why it matters: a system runs mixed versions during every rollout. If a new required field breaks old readers, the deploy takes the fleet down — schema compatibility is what makes zero-downtime releases possible.

Pagination strategies
Offset, cursor, and keyset
Critical

Any endpoint returning a list needs pagination. The strategy determines correctness, database performance, and what clients can do (e.g. jump to page N).

Offset pagination: GET /items?offset=100&limit=20. Simple, supports jumping to page N. Problems: (1) expensive on large offsets — the DB scans and discards 100 rows even with an index; (2) row drift — a row inserted on page 1 while a client is reading page 3 causes duplicates or skipped rows.

Cursor pagination: GET /items?cursor=<opaque_token>&limit=20. The cursor encodes position (last-seen ID or timestamp), opaque to the client. No row drift, O(1) seek with a DB index. Cannot jump to an arbitrary page — next/previous only. Best for feeds, timelines, infinite scroll.

Keyset pagination: GET /items?after_id=1234&limit=20 maps to WHERE id > 1234 ORDER BY id LIMIT 20. Like cursor but the position value is exposed (simpler to implement). Same limitation: no random page access.

Offset (expensive): SELECT * FROM items ORDER BY id LIMIT 20 OFFSET 10000 → scans & discards 10k rows even with an index Keyset (fast): SELECT * FROM items WHERE id > 10000 ORDER BY id LIMIT 20 → index seek directly to id=10000, reads 20 rows — O(log N)

Decision: admin UI needing "jump to page 50" → offset (accept the scan cost, data is small enough). Feed/timeline/infinite scroll at scale → cursor or keyset. Cursor is preferred when you want to hide DB key internals; keyset when simplicity matters more than opacity.

Auth patterns
Sessions, JWTs, API keys, OAuth 2.0
Security

Every system design involves an auth decision. Know the trade-offs.

Session tokens (cookies): server issues a random opaque token, stored server-side in DB or Redis, looked up on every request. Instantly revocable, simple to reason about. Extra lookup per request — mitigated by Redis. Standard for server-rendered web apps and SPAs with a same-origin API.

JWTs (JSON Web Tokens): self-contained signed token carrying claims (user_id, roles, expiry). Stateless — verify the signature without a DB lookup. Risk: cannot be revoked before expiry → use short TTLs (5–15 min) plus a longer-lived refresh token. Good for stateless microservices and cross-domain auth.

API keys: long-lived random strings for server-to-server or B2B calls. No user context. Store only a hash (SHA-256) in the DB — show the raw key once at creation time only. Revoke by deleting the hash.

OAuth 2.0: a delegation protocol — grant a third-party app scoped access to a user's resources without sharing their password. Issues short-lived access tokens + long-lived refresh tokens. (Several grant flows exist for web/mobile vs machine-to-machine; know they exist, not the names.)

Storage engines: LSM vs B-tree
Why your DB choice has a write/read bias
Internals

B-tree (PostgreSQL, MySQL/InnoDB): updates happen in-place on a balanced tree on disk. Great for reads and range scans. Writes do random I/O — you have to find the right page and modify it in place — plus WAL overhead. Fundamentally read-optimized.

LSM-tree (Cassandra, RocksDB, HBase, LevelDB): writes go into an in-memory buffer (memtable), get flushed sequentially to immutable SSTables on disk, and periodically merged via background compaction. Sequential writes means much better write throughput. Reads are slower because you may have to check multiple SSTables — mitigated by Bloom filters that skip SSTables which definitely don't contain the key.

Amplification = extra work vs. the logical op. Write amp = bytes written ÷ bytes you wrote (burns SSD). Read amp = disk reads per lookup. Space amp = disk used ÷ logical size (stale data).

Why they differ: an LSM flushes writes sequentially (cheap ingest), but background compaction rewrites data to merge levels (write amp) and a key sits across several levels until then (read + space amp; Bloom filters soften reads). A B-tree updates a page in place — steady write amp, and a read walks to a single leaf (low read amp).

The practical rule: write-heavy workloads (time-series, logs, high-throughput ingest) → LSM. Read-heavy with complex queries and range scans → B-tree. This is the actual reason to pick Cassandra over Postgres — not "NoSQL is more scalable."

Batch & stream

Moving, transforming, and aggregating data at scale. Relevant for analytics, ML feature engineering, and event-driven architectures. The central question: do you need results in milliseconds or is minutes/hours acceptable?

Batch vs stream processing
Bounded vs unbounded data
Core

Batch processing: operate on a bounded, finite dataset at scheduled intervals (hourly, nightly). High throughput, high latency (results only available after the batch completes). Simple to reason about — the data is static during processing. Tools: Apache Spark, dbt, Hadoop MapReduce.

Stream processing: operate on an unbounded, continuous stream as events arrive. Low latency (milliseconds to seconds). Harder to reason about (out-of-order events, exactly-once semantics, windowing). Tools: Apache Flink, Kafka Streams, Spark Structured Streaming.

Micro-batch vs true streaming (the common follow-up): Spark Structured Streaming processes events in small micro-batches (~seconds), trading latency for Spark's batch engine and simpler semantics. Flink is a true per-event engine — lower latency, native event-time/windowing. That's the main reason to reach for Flink over Spark when sub-second matters.

How stream engines get exactly-once: not by delivering each message once (impossible), but by checkpointing — Flink injects barriers through the dataflow to snapshot all operator state consistently (Chandy-Lamport distributed snapshot), and commits sink output via two-phase commit. On failure it restores the last checkpoint and replays from the saved source offsets → exactly-once processing.

Stream windowing & late data
Aggregating over time in an unbounded stream
Streaming

Stream processors aggregate events into time windows (e.g. "count clicks per minute"). Three window types:

Tumbling windows: fixed-size, non-overlapping. [0s,60s), [60s,120s)…. Each event belongs to exactly one window. Simple, low overhead. Good for per-minute/per-hour aggregations.

Sliding windows: fixed size, slides by a step smaller than the window. "Last 60s, computed every 10s." Overlapping — an event appears in multiple windows. More compute but more granular.

Session windows: gap-based. A window opens on activity and closes when there's been no activity for N seconds. Adapts to user behavior naturally. Good for clickstreams, session analytics.

Late data & watermarks: events arrive out of order (network delay, mobile clients syncing offline). A watermark is a threshold: "I'll wait up to 10 seconds for late events, then close the window and emit results." Late events after the watermark can be dropped, sent to a side output, or used to retract and update the prior result.

Model serving & inference systems
Distributing ML predictions at low latency and high throughput
ML infra

Model serving is a distributed systems problem: serve predictions at low latency, high throughput, to many clients, with multiple model versions deployed safely.

Stateless serving: model weights are read-only once loaded into memory. Any replica can serve any request → scale horizontally behind a load balancer. Deploy new model versions via blue/green (swap 100%) or canary (route X% of traffic, validate metrics, then full rollout).

Dynamic batching: GPUs achieve much higher utilization processing many examples together than one at a time. Collect incoming requests for a short window (e.g. 5 ms), batch them, run one GPU forward pass, fan responses back. Key tuning knob: batch size vs timeout — larger batches = higher throughput but higher latency per request.

Caching predictions: cache model outputs keyed by a hash of the input. Highly effective for repeated queries (search autocomplete, recommendations for popular items). Use a Bloom filter to skip cache for novel inputs without a cache-miss lookup.

Hardware constraints: GPU memory is finite — a large model may exceed a single GPU. Solutions (in complexity order): quantization (use int8/fp16 instead of fp32, shrinks memory ~2–4×), tensor parallelism (split individual weight matrices across GPUs), pipeline parallelism (different layers on different GPUs). In an interview: name these options and state the trade-off (more complexity for larger model support). Don't go deeper unless asked.

Interview framing: treat the model as a black box (input tensor in, output tensor out). The interesting problems are identical to any serving system: how do you scale the fleet, deploy new versions without downtime, handle hardware OOM, and manage latency vs throughput? Training pipelines are DAGs: data validation → feature engineering → training → evaluation → push to serving. Each step produces artifacts (datasets, model weights, metrics) consumed by the next — making this an orchestration and artifact versioning problem.

Experiment tracking: every training run logs hyperparameters, metrics (loss curves, accuracy), and output artifacts (model checkpoints). Enables reproducibility (re-run any experiment), comparison (which config worked best), and lineage (which data version produced which model). Tools: MLflow, Weights & Biases, Vertex AI Experiments.

Distributed training: when model or dataset exceeds one machine. Data parallelism: each worker has a full copy of the model, trains on a different data shard, and gradients are aggregated (AllReduce). Model parallelism: split the model itself across GPUs when it won't fit — either tensor parallelism (split a single layer's weight matrices) or pipeline parallelism (put different layers on different GPUs). Used for very large models (LLMs). Data parallelism is the default; model parallelism adds communication overhead and is harder to implement.

Checkpointing (fault tolerance): long training jobs (hours/days) must periodically save model state to object storage. On a worker crash or preemption, resume from the last checkpoint rather than restarting from scratch. This is the same pattern as write-ahead logging — periodic durable snapshots enable recovery.

Distributed cache. Design a Redis/Memcached-style cache service used by many application services. Support get/set/delete, TTLs, horizontal scale, node failures, hot keys, and cache stampede protection.
Tests: cache-aside role, consistent hashing, TTL/eviction, replication/failover, hot-key and stampede handling.

Scope: this is a cache, not the source of truth. Start with GET key, SET key value ttl, DELETE key, optional MGET, and a client library or proxy that routes keys to cache nodes.

Internal API (gRPC): the cache is a service-to-service dependency, so a typed proto over HTTP/2 fits — compact, low-latency, fast-failing.

syntax = "proto3"; package cache.v1; message CacheEntry { string key = 1; bytes value = 2; // opaque blob — the cache never parses it int64 version = 3; // 0 = don't check (no CAS) } message GetRequest { string key = 1; } message GetResponse { CacheEntry entry = 1; bool found = 2; } // found=false → miss, not an error message MGetRequest { repeated string keys = 1; } message MGetResponse { map<string, CacheEntry> entries = 1; } // absent key = miss message SetRequest { CacheEntry entry = 1; int32 ttl_seconds = 2; } // 0 = server default TTL message SetResponse { bool accepted = 1; string reason = 2; } // accepted=false → stale version (CAS) message DeleteRequest { string key = 1; } message DeleteResponse { bool deleted = 1; } // deleted=false → wasn't present, not an error service CacheService { rpc Get (GetRequest) returns (GetResponse); rpc MGet (MGetRequest) returns (MGetResponse); rpc Set (SetRequest) returns (SetResponse); rpc Delete (DeleteRequest) returns (DeleteResponse); }

Why this shape: value is bytes because a cache is type-agnostic — the app serializes, the cache stores an opaque blob. A miss is a normal result, never a gRPC error: found=false, an absent map entry, or deleted=false. MGet batches reads to amortize round trips. Unlike the payment proto, no idempotency_key is needed — Get is safe and Set/Delete are naturally idempotent.

version = compare-and-swap, and only cache-aside needs it. It closes the populate-on-miss race: a reader that fetched an old DB value is rejected when it tries to Set a version older than what's cached. Write-through/write-back populate on the write path and never read-then-populate, so they have no such race. Prefer ttl_seconds = 0 → server default over "no expiry" — an unset proto3 int is also 0, and silently-permanent keys leak memory.

Architecture: use consistent hashing with virtual nodes (or fixed slots like Redis Cluster) so adding/removing nodes only moves a fraction of keys. Store values in memory with TTL plus LRU/LFU eviction. Replicate hot/important shards if availability matters.

Writes and consistency: most product systems use cache-aside: app writes the database, then invalidates the cache key. Do not update the cache first and pretend it is durable. Stale reads are acceptable only within the TTL/invalidation window.

Deep dives: hot key gets replicated or locally cached; node failure causes misses and refill, not data loss; stampede is handled with single-flight, jittered TTLs, stale-while-revalidate, cache warming (pre-warm hot keys on a cold cache or before a spike), and backpressure to protect the origin DB.

Trap: saying "just add Redis" without explaining key routing, rebalancing, eviction, and what happens when a popular key expires.

Search engine (indexing + serving). Design the search layer that answers full-text queries over a crawled corpus of billions of documents: index the documents, return ranked results in under 200ms, paginate, and keep new pages searchable.
Tests: inverted index, document vs term partitioning, scatter-gather + tail latency, ranking (BM25), index freshness.

Scope: this is the layer after the crawler. Accept a query, return ranked document results with pagination, and make newly-crawled docs searchable. Crawling/ingestion is upstream; deep ML ranking is mostly out of scope beyond a solid baseline.

The inverted index is the core. A search engine never scans documents at query time — it stores term → sorted posting list of doc ids (each posting carrying term frequency + positions). A query intersects (AND) or merges (OR) those posting lists, doing work proportional to matching docs, not corpus size.

Architecture: an indexer tokenizes/normalizes each doc (lowercase, stem, drop stopwords) and appends it to each term's postings, written as immutable segments that merge in the background. A query coordinator scatters the query to every index shard; each shard returns a local top-k from its postings; the coordinator gathers, merges, and re-ranks (BM25) before hydrating title/url/snippet. A result cache absorbs hot queries.

Data model: documents(doc_id, url, title, body, lang, fetched_at); the inverted index term → [(doc_id, tf, positions)] stored as immutable segments; a per-shard doc store for hydration.

Deep dives: partition by document (each shard indexes a slice; every query scatter-gathers) rather than by term — the default, since term partitioning makes multi-term queries join across hot shards. Scatter-gather makes the slowest shard your latency, so replicate shards and use hedged requests, per-shard timeouts, and partial results. Freshness comes from a small in-memory segment that's immediately searchable, merged later. Rank with BM25 (tf, idf, doc-length), optionally re-ranking just the top-k with a learned model.

Trap: scanning documents (or a SQL LIKE '%q%') instead of an inverted index, a single un-sharded index, or forgetting that scatter-gather means your p99 is the slowest shard.

DynamoDB vs Bigtable, the one-liner: DynamoDB gives you a sort key and GSIs, so a new access pattern is usually a new GSI. Both punish unplanned patterns; Bigtable just makes the cost explicit.

Same problem in Cloud Spanner (relational, globally distributed). Spanner gives you SQL, transactions, and secondary indexes — and it lets you physically co-locate parent and child rows with INTERLEAVE IN PARENT, so an order and its items live in the same split and join without a cross-node hop.

Same data in BigQuery (analytical warehouse). BigQuery is columnar and built for scans over the whole dataset, not low-latency point reads — so you flip the model: denormalize, embed line items as a repeated field, and prune cost with partitioning + clustering instead of indexes.

CREATE TABLE orders ( order_id STRING, user_id STRING, created_at TIMESTAMP, status STRING, total NUMERIC, items ARRAY<STRUCT<product_id STRING, qty INT64, price NUMERIC>> -- line items nested in the row ) PARTITION BY DATE(created_at) -- prune by day CLUSTER BY user_id, status; -- co-locate a user's / a status's rows

(a) profiles don't belong here — point reads are an OLTP job (keep them in DynamoDB/Spanner); BigQuery holds the analytics copy.

(b) SELECT * FROM orders WHERE user_id=@u ORDER BY created_at DESC — clustering on user_id prunes the scan (seconds, not milliseconds).

(c) line items are in the row: SELECT i.* FROM orders, UNNEST(items) i WHERE order_id=@o — no join.

(d) the product lookup is the query BigQuery is made for: … FROM orders, UNNEST(items) i WHERE i.product_id=@p, pruned by partition + cluster.

The BigQuery trap: cost is bytes scanned, not rows returned. SELECT * on an unpartitioned table reads everything; always partition on the time column and select only the columns you need. And keep it off the serving path — it's an analytics store, not a key-value DB.
Four stores, one access-pattern list — the through-line: DynamoDB → sort key + GSIs; Bigtable → one row key, you dual-write your own "index"; Spanner → SQL + interleaving + real secondary indexes; BigQuery → denormalize + partition/cluster for scans, no serving-path reads. Pick the store from the access patterns, then design keys to match — not the other way round.

Claim a job — SELECT … WHERE status='queued' AND run_at <= now() ORDER BY priority DESC, run_at FOR UPDATE SKIP LOCKED LIMIT N, then set status='running', lease_token=<uuid>, locked_until=now()+timeout. SKIP LOCKED stops workers colliding without serializing the whole pool.

Lease / visibility timeout — a crashed worker's locked_until expires, so a sweeper or claim query makes it claimable again. You model a lease column; you do not hold a DB transaction open while the job runs.

Retry + dead-letter — on failure, if attempts < max_attempts set status='queued' and run_at = now()+backoff+jitter; otherwise status='dead' (the DLQ) with last_error for replay/debugging.
